# SideChat 打包发布流程

本文档是 SideChat 的发布清单。以后如果对 Codex 说“部署发布”或“编译打包发布”，默认按这里从版本号、编译、打包、提交、tag 到 GitHub Release 一次性完成。

## 发布目标

- 生成可安装的 VSIX：`sidechat-<version>-<YYYYMMDD>.vsix`
- 将源码改动、版本号改动、VSIX 产物提交到 `main`
- 打 annotated tag：`v<version>`
- 推送 `main` 和 tag 到 GitHub
- 创建 GitHub Release，并把 VSIX 作为 release asset 上传

## 完整发布流程

### 1. 发布前检查

```bash
git status -sb
git log --oneline --decorate -5
gh --version
gh auth status
```

确认：

- 当前仓库是 `kyruntime/kkyy-side-chat`
- 当前分支是要发布的分支，通常是 `main`
- `gh auth status` 已登录，并且 token 有 `repo` 权限
- 工作区没有不属于本次发布的混杂改动

### 2. 递增版本号

打包前必须递增 `package.json` 里的 `version`，并同步 `package-lock.json` 顶部和 root package 里的版本号。

示例：

```json
"version": "1.8.4"
```

Cursor 根据版本号判断是否需要更新扩展，版本号不变则不会识别为新版。

### 3. 编译和基础校验

```bash
npm run compile
node --check mcp-server/index.mjs
git diff --check
```

说明：

- `src/` 是源码目录，`dist/` 是编译产物，扩展运行时加载的是 `dist/`
- 改完 `src/` 后必须执行 `npm run compile`
- `mcp-server/` 是独立 MCP 服务端，不经过 TypeScript 编译，所以要单独 `node --check`
- `git diff --check` 用来发现行尾空格等容易污染提交的小问题

### 4. 打包 VSIX

```bash
npm run package
VERSION="$(node -p "require('./package.json').version")"
DATE="$(date +%Y%m%d)"
mv "sidechat-${VERSION}.vsix" "sidechat-${VERSION}-${DATE}.vsix"
```

如果 `npm run package` 提示 `package.json` 缺少 `repository` 字段，这只是 warning，不影响当前 VSIX 生成。

### 5. 校验 VSIX

```bash
VERSION="$(node -p "require('./package.json').version")"
DATE="$(date +%Y%m%d)"
VSIX="sidechat-${VERSION}-${DATE}.vsix"

ls -lh "$VSIX"
shasum -a 256 "$VSIX"
unzip -p "$VSIX" extension/package.json \
  | node -e "let s='';process.stdin.on('data',d=>s+=d);process.stdin.on('end',()=>{const p=JSON.parse(s);console.log(p.name,p.version,p.main)})"
```

确认：

- VSIX 文件存在
- VSIX 内部版本号等于 `package.json` 版本号
- `main` 是 `./dist/extension.js`
- 记录 SHA256，后续写入 GitHub Release notes

### 6. 提交源码和版本号改动

```bash
git add mcp-server/index.mjs package.json package-lock.json src/extension.js src/session.js
git status --short
git commit -m "bump sidechat to ${VERSION}"
git push origin main
```

实际 staging 文件应按本次改动调整，不要无脑 `git add -A`。如果只有文档或其它文件，也只 stage 本次发布需要的文件。

### 7. 提交 VSIX 产物

仓库 `.gitignore` 默认忽略 `*.vsix`，发布包需要强制加入：

```bash
git add -f "$VSIX"
git status --short
git commit -m "release sidechat ${VERSION} vsix"
```

推荐把 VSIX 放在单独提交里，这样发布产物提交清晰可查。

### 8. 打 tag 并推送

```bash
TAG="v${VERSION}"

git tag -a "$TAG" -m "SideChat ${VERSION}"
git show --stat --oneline --decorate "$TAG"
git push origin main
git push origin "$TAG"
```

tag 必须打在包含 VSIX 的提交上，这样 `v<version>` 对应的源码和安装包在同一个快照里。

### 9. 创建 GitHub Release 并上传附件

```bash
SHA256="$(shasum -a 256 "$VSIX" | awk '{print $1}')"

gh release create "$TAG" "$VSIX" \
  --repo kyruntime/kkyy-side-chat \
  --title "SideChat ${TAG}" \
  --notes "## Changes
- Describe the user-facing changes in this release.

## Asset
- ${VSIX}

SHA256: ${SHA256}" \
  --latest
```

如果 release 已存在但缺少附件，用：

```bash
gh release upload "$TAG" "$VSIX" \
  --repo kyruntime/kkyy-side-chat \
  --clobber
```

### 10. 发布后确认

```bash
gh release view "$TAG" \
  --repo kyruntime/kkyy-side-chat \
  --json tagName,name,url,assets,isDraft,isPrerelease,publishedAt

git status -sb
git log --oneline --decorate -3
```

确认：

- GitHub Release URL 可打开
- release asset 中有 `sidechat-<version>-<date>.vsix`
- asset 状态是 `uploaded`
- release 不是 draft，也不是 prerelease，除非本次明确要求
- 本地 `main` 和 `origin/main` 对齐

## 安装测试

```bash
# 在 Cursor 中安装 VSIX
# 方式一：命令面板 -> Extensions: Install from VSIX...
# 方式二：
code --install-extension sidechat-x.x.x-xxxxxxxx.vsix
```

安装后打开 SideChat 侧栏，至少确认：

- 侧栏可以加载
- 工作区配置可以写入 `.cursor/mcp.json`
- `check_messages` 能在 Cursor Agent 中正常运行
- 本次修改涉及的功能表现符合预期

## Codex 执行约定

当用户说“部署发布”“编译打包发布”“发一个新版本”时，Codex 默认执行完整发布流程：

1. 检查 git 状态、远端、`gh` 登录状态
2. 递增 patch 版本号，除非用户指定版本号
3. 编译并校验
4. 打包并重命名 VSIX
5. 校验 VSIX 版本和 SHA256
6. 提交源码和版本号改动
7. 强制加入并提交 VSIX
8. 打 `v<version>` annotated tag
9. 推送 `main` 和 tag
10. 创建 GitHub Release，上传 VSIX，并标记 latest
11. 回报 commit、tag、release URL、VSIX URL 和 SHA256

如果工作区有不属于本次发布的混杂改动，必须先说明并确认提交范围，不能静默一起提交。

## 注意事项

- `src/` 是源码目录，`dist/` 是编译产物，扩展运行时加载的是 `dist/`
- `mcp-server/` 是独立 MCP 服务端，直接打包进 VSIX
- 打包命令 `vsce package --no-dependencies` 不会把 `node_modules` 打进去，安装时由扩展自行 `npm install`
- VSIX 文件虽然被 `.gitignore` 忽略，但正式发布时需要 `git add -f`
- GitHub 右侧 Releases 不会因为 push tag 自动更新为最新 release，必须创建 GitHub Release
